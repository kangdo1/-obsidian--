# AI Hallucination 검증 프로세스

> 목적: AI가 생성한 보고서·문서의 사실관계를 체계적으로 검증하는 표준 프로세스
> 적용 범위: AI 생성 보고서, 기술 문서, 분석 자료 등 모든 AI 산출물

---

## 프로세스 전체 흐름

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌──────────────┐
│  Phase 0    │    │  Phase 1    │    │  Phase 2    │    │  Phase 3    │    │  Phase 4     │
│  AI 산출물  │───▶│  출처 검증  │───▶│  교차 검증  │───▶│  전문가     │───▶│  최종 반영   │
│  생성       │    │             │    │             │    │  검토       │    │  및 기록     │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘    └──────────────┘
                         │                  │                  │                    │
                    출처 있음/없음     정확/오류/불확실    확인/수정/보류      수정본 확정
```

## Phase 0: AI 산출물 생성

### 0-1. 프롬프트 작성 (RCIF 프레임워크)

| 요소 | 작성 포인트 |
|------|-----------|
| **R (역할)** | 구체적 전문가 역할 부여 → 품질 기준을 높임 |
| **C (맥락)** | 대상 독자, 배경, 원본 데이터를 충분히 제공 |
| **I (지시)** | 포함할 항목을 번호 리스트로 명확히 지정 |
| **F (형식)** | 출력 템플릿을 제시하여 일관된 결과물 확보 |

### 0-2. 검증 대비 요청사항

AI에게 보고서 생성 시 다음을 함께 요청한다:

```
- 모든 수치 데이터에 출처를 명시해주세요
- 확실하지 않은 정보는 "추정", "미확인"으로 표시해주세요
- 서로 다른 모델/버전의 사양은 명확히 구분해주세요
```

> 이렇게 요청하면 AI가 스스로 불확실한 부분을 표시하여 검증 범위를 줄일 수 있다.

## Phase 1: 출처 검증

### 1-1. 출처 분류

보고서에 명시된 모든 출처를 다음 기준으로 분류한다:

| 등급 | 기준 | 예시 | 신뢰도 |
|------|------|------|--------|
| **A등급** | 1차 출처 (원본 데이터) | 정부 통계, 제조사 공식 사양서, 학술 논문 | 높음 |
| **B등급** | 2차 출처 (가공·분석) | 연구기관 보고서, 업계 전문지 기사 | 보통 |
| **C등급** | 3차 출처 (재인용) | 블로그, 위키, 일반 뉴스 기사 | 낮음 |
| **없음** | 출처 미명시 | - | 검증 필수 |

### 1-2. 출처 존재 확인

| 점검 항목 | 확인 방법 |
|----------|----------|
| 출처 문서가 실제 존재하는가? | 해당 기관 웹사이트에서 직접 검색 |
| 출처명이 정확한가? | 보고서명, 발행연도, 저자 확인 |
| AI가 출처를 날조하지 않았는가? | 존재하지 않는 논문·보고서를 지어내는 경우가 있음 |
| 출처의 내용이 인용과 일치하는가? | 원문을 직접 읽어 확인 |

### 1-3. 출처 검증 결과 기록

```markdown
| 출처 | 존재 여부 | 등급 | 비고 |
|------|----------|------|------|
| (출처명) | ✅ / ❌ / ⚠️ | A/B/C | (상세) |
```

**판정 기준:**
- ✅ 확인: 출처가 존재하고 인용 내용이 일치
- ⚠️ 모호: 출처는 있으나 구체적 문서 특정 불가
- ❌ 미확인: 출처가 존재하지 않거나 확인 불가 → **Phase 2에서 반드시 교차 검증**

## Phase 2: 교차 검증

### 2-1. 검증 대상 선정

모든 내용을 검증할 수 없으므로, **우선순위**에 따라 검증 대상을 선정한다:

| 우선순위 | 대상 | 이유 |
|---------|------|------|
| **최우선** | 구체적 수치 (출력, 효율, 비율, 통계) | AI가 수치를 혼동하거나 날조할 확률이 가장 높음 |
| **높음** | 시점·일자 (실증 완료일, 제도 시행일) | 과거/미래 시점 혼동 빈발 |
| **높음** | 고유명사 (모델명, 기업명, 프로젝트명) | 명칭 변경, 유사 이름 혼동 |
| **보통** | 인과관계, 비교 서술 | 논리적 오류 가능성 |
| **낮음** | 일반적 개념 설명, 원리 서술 | 대체로 정확하나 확인 권장 |

### 2-2. 검증 방법

**원칙: 최소 2개의 독립적 출처에서 확인**

| 검증 방법 | 적합한 대상 | 도구 |
|----------|-----------|------|
| **제조사 공식 사양서** | 장비 출력, 효율, 사양 | 제조사 웹사이트 |
| **정부·공공기관 통계** | 설비용량, 발전비중, 정책 | e-나라지표, 에너지공단, 전력거래소 |
| **학술 논문 DB** | 기술 데이터, 연구 결과 | Google Scholar, DBpia, RISS |
| **업계 전문지** | 프로젝트 현황, 기업 동향 | Power Magazine, 전기신문, 에너지경제 |
| **AI 교차 질문** | 1차 확인 용도 | 다른 AI 모델에 같은 질문 |

### 2-3. 흔한 Hallucination 패턴 체크리스트

AI가 자주 만드는 오류 유형을 사전에 점검한다:

| # | 패턴 | 설명 | 점검 방법 |
|---|------|------|----------|
| 1 | **모델/세대 혼동** | 구형과 신형, 기본형과 개량형의 사양을 섞어서 기재 | 모델명별 사양을 개별 확인 |
| 2 | **목표 → 실적 변환** | 개발 목표치를 이미 달성한 것처럼 서술 | "목표", "계획", "예정" 등의 표현 확인 |
| 3 | **시연 → 상업운전 과대표현** | 일회성 테스트를 지속적 상업 운전으로 표현 | 보도자료에서 "demonstration", "test" 여부 확인 |
| 4 | **미래 → 과거 혼동** | 아직 일어나지 않은 일을 완료형으로 서술 | 날짜·시점을 현재 날짜와 비교 |
| 5 | **출처 날조** | 존재하지 않는 논문, 보고서, 통계를 만들어냄 | 출처 URL 직접 접속, 논문 DB 검색 |
| 6 | **수치 근사치 날조** | 그럴듯한 숫자를 만들어냄 (정확해 보이지만 출처 없음) | 공식 통계와 대조 |
| 7 | **기업명/기관명 오류** | 사명 변경, 합병, 분사를 반영하지 못함 | 최신 기업 정보 확인 |

### 2-4. 교차 검증 결과 기록

```markdown
### 발견된 오류
| # | 항목 | 보고서 기재 | 실제 확인 | 패턴 | 검증 출처 |
|---|------|-----------|----------|------|----------|

### 검증 통과
| 항목 | 보고서 내용 | 판정 |
|------|-----------|------|
```

## Phase 3: 전문가 검토

### 3-1. 전문가 검토가 필요한 경우

| 상황 | 이유 |
|------|------|
| 웹 검색으로 검증 불가능한 수치 | 비공개 데이터, 최신 미발표 자료 |
| 해석이 필요한 기술적 판단 | 단순 사실 확인이 아닌 전문적 해석 |
| 정책·제도의 세부 내용 | 시행령, 고시 등 세부 규정 확인 |
| 보고서의 결론·제언이 합리적인지 | 논리적 타당성 판단 |

### 3-2. 전문가 검토 요청 방법

```
검토 요청서
─────────────────────────────
1. 검토 대상: (보고서 제목)
2. 검토 항목: (구체적 질문 리스트)
3. 배경: AI가 생성한 초안이며, 팩트체크 중 확인이 어려운 항목입니다.
4. 요청 사항: 아래 항목의 수치/사실관계 확인을 부탁드립니다.
   - 항목 1: ...
   - 항목 2: ...
5. 참고 자료: (보고서 해당 섹션 발췌)
─────────────────────────────
```

### 3-3. 분야별 확인 기관 예시

| 분야 | 확인 기관/출처 |
|------|--------------|
| 국내 전력 통계 | 한국전력거래소(KPX), 전력통계정보시스템(EPSIS) |
| 재생에너지 보급 | 한국에너지공단(KEA), 신재생에너지센터 |
| 에너지 정책 | 산업통상자원부, 에너지경제연구원(KEEI) |
| 가스터빈 기술 | 각 제조사 기술영업팀, 한국기계연구원(KIMM) |
| 전력시장 제도 | 전력거래소 시장운영실, 에너지경제연구원 |

## Phase 4: 최종 반영 및 기록

### 4-1. 수정 반영

| 작업 | 내용 |
|------|------|
| 오류 수정 | Phase 2에서 확인된 Hallucination을 정확한 정보로 교체 |
| 출처 보강 | 검증에 사용한 출처를 보고서에 추가 (URL 포함) |
| 불확실 표시 | Phase 3 대상 항목에 "전문가 확인 필요" 주석 추가 |
| 검증 이력 표시 | 보고서 상단에 검증 완료 표시 및 검증일 기재 |

### 4-2. 검증 결과 문서 작성

검증 과정과 결과를 별도 문서로 기록한다:

```markdown
# (보고서명) - Hallucination 검증 결과

> 검증일:
> 검증 대상:
> 검증자:

## 종합 평가

| 구분 | 건수 |
|------|------|
| ✅ 정확 | N건 |
| ⚠️ 오류 (수정 완료) | N건 |
| 🔍 전문가 확인 필요 | N건 |

**신뢰도 등급**: 높음 / 보통 / 주의 필요

## 주요 발견 사항
- ...

## 수정 이력
| # | 수정 항목 | 수정 전 | 수정 후 | 검증 출처 |
|---|----------|--------|--------|----------|
```

### 4-3. 신뢰도 등급 기준

| 등급 | 기준 | 활용 권장 |
|------|------|----------|
| **높음** | 오류 0~1건, 전문가 확인 필요 2건 이하 | 공식 문서로 사용 가능 |
| **보통** | 오류 2~4건, 대부분 수치 오류 | 수정 후 내부 참고 자료로 활용 |
| **주의 필요** | 오류 5건 이상 또는 핵심 사실관계 오류 | 전면 재검증 후 사용 |

## 빠른 참조: 검증 체크리스트

실무에서 빠르게 사용할 수 있는 간소화 버전:

- [ ] **출처 확인**: 모든 주요 주장에 출처가 있는가?
- [ ] **출처 존재**: 명시된 출처가 실제 존재하는가?
- [ ] **수치 검증**: 핵심 수치를 2개 이상 독립 출처로 확인했는가?
- [ ] **시점 확인**: 날짜·시점이 현재와 맞는가? 미래를 과거로 쓰지 않았는가?
- [ ] **모델 구분**: 서로 다른 모델/버전의 사양이 혼동되지 않았는가?
- [ ] **표현 수준**: 시연(Demo)을 상업운전으로, 목표를 실적으로 과대표현하지 않았는가?
- [ ] **고유명사**: 기업명, 모델명, 프로젝트명이 최신 정보와 일치하는가?
- [ ] **전문가 확인**: 검증 불가 항목을 표시하고 전문가 검토를 요청했는가?
- [ ] **수정 반영**: 발견된 오류를 보고서에 반영하고 출처를 추가했는가?
- [ ] **이력 기록**: 검증 결과를 별도 문서로 기록했는가?
